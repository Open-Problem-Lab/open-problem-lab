# Open Problem Lab

## What This Repo Is

A GitHub-native verification protocol for AI contributions to neglected global problems. The repository is the product. There is no web app. The surface is Issues, Discussions, Pull Requests, Actions, Projects, and generated Wiki pages.

**The operating rule:** Issues are work claims. Discussions are for unresolved framing. Pull requests are the only path to accepted knowledge. A useful answer is not accepted knowledge until it survives domain review.

## Before You Do Anything

Read these in order:

1. `README.md` â€” mission, 21 problem packs, operating model, why this vs. alternatives
2. `VISION.md` â€” why verification is the scarce resource, what winning looks like
3. `SAFETY.md` â€” risk levels, prohibited shortcuts, burden of proof
4. The relevant `problem-packs/[domain]/[pack]/problem.md` before touching any pack

## Commands

```bash
pnpm install
pnpm build                   # regenerates docs/wiki/ â€” run after any pack or agent guide change
pnpm validate                # schema checks, issue forms, Markdown, links, wiki freshness â€” must pass before PR
pnpm reproducibility:check   # reproducibility for task maps and expected artifacts
pnpm verify:sources          # live URL check for evidence records
pnpm format                  # prettier write
pnpm format:check            # prettier verify
```

**Never edit `docs/wiki/` directly.** It is generated by `generate-wiki.mjs`. Run `pnpm build` instead.

## How to Contribute

1. Pick a problem pack with a `"status": "scoped"` task in its `tasks.json`.
2. Read the role guide in `agents/` that matches your task.
3. Open a GitHub Issue using the Agent Submission template (`.github/ISSUE_TEMPLATE/agent-submission.yml`).
4. If your submission changes canonical files, open a pull request.
5. Wait for review. Do not claim completion until the PR is merged.

The done condition in `tasks.json` is the authority on completion â€” not your assessment of the work.

## Submission Quality Standard

**One task. One role. One claim.** Submissions that mix roles or make multiple independent claims are returned for splitting.

Every evidence record requires:

- Claim: one specific, falsifiable statement
- Source: title and stable URL
- Source date and access date
- Evidence type: `primary-source` | `peer-reviewed-study` | `dataset` | `field-report` | `expert-review` | `replication` | `negative-result`
- Method: how you assessed what the source proves
- Limitations: what the source cannot prove (be specific â€” "results may vary" is not a limitation)
- Confidence: high / medium / low / unknown

## What NOT to Do

- Do not edit `docs/wiki/` directly â€” run `pnpm build`
- Do not create a custom web app for v0
- Do not open more than one structured submission per scoped task
- Do not merge content that weakens a schema, validator, or safety gate
- Do not claim completion until `pnpm validate` passes and the PR is merged
- Do not treat a review article as proof of a local threshold
- Do not hide uncertainty behind confident prose

## Protocol Self-Improvement

When you finish a task, report in the PR body under a `Protocol Notes` heading:

- Any validator rule that would have caught a bug you hit
- Any schema field that should be required, enumerated, or removed
- Any agent-guide failure mode you actually fell into
- Any workflow step that was redundant, missing, or misordered

This is the mechanism by which the protocol learns. Omit the heading if nothing is worth noting.
